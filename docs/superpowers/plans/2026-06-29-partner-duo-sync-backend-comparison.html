Partner Duo Sync — Backend Options

Making both parents first-class writers ("a duo"), assuming Firebase is replaced. Architectural options, trade-offs, and current pricing for each candidate backend.

Akachan / BabyTracker · Generated 2026-06-29 · Companion to the Firebase → Supabase migration plan

Today the primary device owns all shared data (Room) and the partner only submits operation documents the primary later applies. We want both devices to execute actions directly — start a nap, log a bottle — with live, two-way updates. This report compares backends that can deliver that once Firebase is dropped.

The key reframing. The current op-inbox (feedOps/sleepOps → primary applies → re-publishes snapshot) was largely a workaround for Firestore: a single snapshot doc with a 1 MB ceiling, whole-field last-write-wins, and no clean way for two devices to write the same collection without echo loops. On a Postgres backend, that whole problem class disappears. The natural model becomes: shared rows, both devices write, row-level security gates by share membership (not "owner"), and a realtime stream pushes changes to the other phone. Symmetric authority falls out of the platform — there's no privileged executor to port.

So the real decision is no longer "how do I let the partner write." It's how much of the offline / conflict sync do I build versus buy. That axis orders the four options below.

One fix needed in every option: identity. Anonymous auth that resets on reinstall strands a parent's authorship and share access. In a symmetric duo there is no "primary" to re-bootstrap from, so lightweight real accounts (or device-key linking) in the chosen auth system matters more here than it did before. Budget for it regardless of which option wins.

Current architecture (what we're replacing)

DirectionMechanism today
Partner → PrimaryPartner writes op docs to shares/{code}/feedOps · sleepOps. Never touches shared state directly.
Primary appliesForeground Firestore listener + ~15-min PartnerOpDrainWorker → applies ops into Room → re-publishes snapshot → deletes ops.
Primary → PartnerPushes the whole snapshot to a single doc shares/{code}.
Partner readsFetches that snapshot, merges its own pending ops for optimistic UI. No local DB of shared data.

Two failure modes drive the "not a duo" feeling: write authority (partner ops are inert — and if the primary phone is off, lost — until the primary processes them) and propagation latency (no push; realtime only while the primary is foregrounded, otherwise a 15-min floor). They are independent; a Postgres + realtime stack fixes both.

The options

Option A · RecommendedSupabase-native, DIY sync

The planned migration, extended for symmetry. Both devices write to shared Postgres tables; RLS gates by share membership; Supabase Realtime (Postgres Changes) streams row changes to the other device; offline writes queue in the Room outbox you're already building and flush on reconnect; conflicts resolve last-write-wins by updated_at with delete tombstones.

  • Fits the roadmap exactly — no vendor beyond the Supabase you're already adopting. The duo becomes mostly an RLS rule (membership, not owner) + both devices subscribing & writing.
  • Op-inbox, snapshot doc, and the 1 MB ceiling are all deleted. No privileged primary.
  • You own the conflict policy; Postgres holds canonical state.
  • You hand-roll the hard part: outbox retry, dedupe, delete tombstones, LWW, and the Realtime↔Room reconciliation — exactly where local-first apps go subtly wrong.
  • Use Realtime's Postgres Changes mode (RLS-respecting), not Broadcast. The per-change RLS check is irrelevant at two devices.

Option BSupabase + a local-first sync engine (PowerSync)

Keep Supabase Postgres, but drop PowerSync between device SQLite and Postgres. You read/write a local DB directly (instant, offline); PowerSync owns the upload queue, reconnect, realtime streaming and conflict handling. Both devices are equal peers. This retires the op-inbox and the outbox you were about to build.

  • Least sync code you write — the offline queue, dedupe and streaming are the product's job. True duo + offline + live with the smallest hand-rolled surface.
  • Has a Kotlin Multiplatform / Android SDK (reactive via Coroutines/Flows), first-class Supabase integration, non-invasive to the Postgres schema. Composes with the migration.
  • Adds a second service + cost on top of Supabase — two things to operate and pay for.
  • The Kotlin SDK is open alpha — real risk for a shipping app; small community.
  • You'd likely replace Room with PowerSync's SQLite for shared data (or run both). The app is Room-centric today — a real splice.
  • Conflict model is LWW at the queue/row level, not CRDT — fine for two parents who rarely edit the same field in the same second.

Option CRealtime transport only — keep a symmetric op-log

Don't adopt table-sync. Use Supabase purely as (1) a Postgres ops table holding the same FeedOp/SleepOp docs you already model and (2) Realtime to deliver them instantly. Both devices write ops and both apply ops into their own Room. It's the current op-inbox, made bidirectional and live, on Postgres.

  • Maximum reuse — FeedOp/SleepOp/ApplyFeedOpUseCase/ReconcilePendingOps survive almost verbatim; swap Firestore for a table + Realtime. Smallest change mid-flight on the Collaborative Partner View work.
  • Live + symmetric immediately.
  • You keep the op-log's accidental complexity: an op must live until both peers apply it, so delete-after-apply becomes per-peer ack cursors; plus server-timestamp ordering and idempotent apply. A hand-maintained mini event-sourcing system.
  • You use a relational DB as a dumb queue and still rebuild state in Room on each device — Option A gets the same duo with Postgres keeping state and two fewer representations per entity.

Best use: an interim step if the migration timeline forces shipping the duo before full table-sync lands.

Option D · SkipTurnkey mobile-sync platform that isn't Supabase

Couchbase Lite + Capella App Services (embedded DB, multi-master replication, built-in conflict resolution, native Kotlin SDK — purpose-built for "N devices sync a shared dataset offline") or AWS Amplify DataStore + AppSync (local-first, versioned conflict resolution, Android SDK — the Firestore analog).

  • Most of the sync problem is solved by the platform; battle-tested for multi-device offline.
  • Contradicts the Supabase decision. Couchbase = NoSQL doc model (re-think the relational schema you're migrating to); Amplify = AWS lock-in + GraphQL codegen + AWS being lukewarm on DataStore.
  • Heavier ops / lock-in for a small app. Only sensible if you abandon Supabase — which is not on the table.

Recommendation

Default to Option A (Supabase-native). You're already building the outbox + tables; making it a duo is largely an RLS change (membership, not owner) plus both sides subscribing. You get symmetric authority and live updates as platform properties and delete a whole subsystem (the op-inbox). The only real risk is the hand-rolled offline/conflict reconciliation.

Reach for Option B (PowerSync) only when that reconciliation bites. It's built precisely to remove that risk and keeps Supabase — but it's alpha and adds a service, so don't buy the engine before you've felt the pain of hand-building it.

Use Option C only as an interim if the timeline forces shipping the duo before full table-sync lands. Skip Option D — it trades away the Supabase decision for a marginal turnkey benefit.

The decision reduces to one risk question: do you trust yourself to get offline LWW + tombstones + reconnect right? Yes → A. "I'd rather pay someone" → B. Either way, fix the anonymous-identity problem as part of it.

Rejected alternatives

  • ElectricSQL — pivoted (v1.1, Aug 2025) to read-path-only sync (Shapes API); writes are bring-your-own back to Postgres. No turnkey bidirectional duo writes anymore. Not a fit.
  • Convex / InstantDB — realtime/offline but web/React-first; native Kotlin/Android support is weak or absent.
  • CRDTs (Automerge / Yjs) — perfect concurrent-merge, but heavy/immature on Kotlin, and two parents almost never edit the same field simultaneously. LWW is enough. Overkill.
