Where should Akachan store data if it goes public?

An evaluation of backend / database options for a local-first Android baby tracker, with cost-at-scale modelling and the trade-offs of each path — Firebase, Supabase, Appwrite, PocketBase, local-first sync engines, and DIY Postgres.

Prepared June 2026 · For the Akachan (BabyTracker) project · Pricing figures are indicative and change frequently — confirm against vendor pages before committing.

TL;DR

The cost fear is real in general — but probably not for your app.

Akachan is local-first: Room (SQLite on-device) is the source of truth and holds all core tracking data. Firebase only stores a small read-only snapshot per family for the optional partner-sharing feature. Your fan-out is 1 — one snapshot is read by one partner. The "Firestore is ruinously expensive at scale" stories almost always come from chat / social / feed apps where one write fans out to thousands of reads. That is a different workload from yours.

Modelled below: even at 100,000 active sharing families, Firestore costs you roughly $50–80/month. Cost is not your real decision driver. The real drivers are vendor lock-in, bill-spike risk from bugs, data residency / GDPR (baby health data), and how much ops work you want to own.

Recommendation in one line

Stay on Firebase now (it is already built, cheap at your scale, and zero-ops). Treat a migration as a later decision triggered by GDPR/residency needs or a desire to kill lock-in — and if you migrate, the architecturally honest target is either Supabase (managed Postgres, predictable pricing, EU region) or a local-first sync engine (PowerSync / Turso) that mirrors how your app already works.

1. First, understand what you're actually paying for

"Database" is the wrong frame for this app. You already have a database — Room/SQLite on the phone — and it costs nothing and scales perfectly because every user runs their own copy. The thing under discussion is the sync / sharing backend: the small cloud layer that lets a partner see a read-only snapshot.

So the comparison is really between backend-as-a-service platforms and sync engines, judged on: cost model, lock-in, control, data residency, and operational burden.

Your workload, sized honestly

A baby generates a handful of tracked events per day (feeds, sleeps, the odd note) — call it ~15 events/day per family. Each event = 1 snapshot write. A partner with a realtime listener sees ~1 read per change. So a sharing pair is roughly ~30 read+write operations per day. This is tiny. Baby tracking is one of the least backend-intensive app categories that exists.

2. The options

Firebase Firestore Current

Google's serverless NoSQL document DB + Auth, realtime listeners, fully managed.

  • Already integrated — anonymous auth + snapshot sync are built and working.
  • Zero ops — no servers, scaling, or patching.
  • Generous free tier — 50k reads / 20k writes / 1 GiB per day; you likely never leave it solo or in early launch.
  • Best-in-class offline + realtime SDK for Android.
  • Pay-per-operation — cost is unpredictable and a bug can spike the bill.
  • Vendor lock-in — proprietary query model; migration is real work.
  • Data residency — region is picked at project creation and is hard to change; weaker fit for strict EU health-data stances.

Supabase Open source

Managed Postgres + Auth + realtime + storage. Open source; self-hostable; EU regions.

  • Predictable, tier-based pricing — Free, then $25/mo Pro per project; unlimited API requests (no per-read billing).
  • Real SQL / Postgres — relational, portable, no lock-in to a proprietary query model.
  • Independent comparisons put it ~3–5× cheaper than Firebase on heavy read/write workloads.
  • EU hosting + self-host option — strong for GDPR / health data.
  • You'd rebuild the sync + auth layer (no anonymous-auth-to-snapshot path out of the box like you have now).
  • Free-tier projects pause after 7 days of no traffic (fine once you have users; annoying pre-launch).
  • Offline/realtime mobile story is less turnkey than Firestore's.

Local-first sync engines Best arch fit

PowerSync, Turso, ElectricSQL — sync a cloud Postgres/SQLite down to on-device SQLite with offline + conflict resolution.

  • Matches what you already do — you're local-first with SQLite/Room today; this formalizes it instead of bolting a foreign cloud DB on the side.
  • True offline-first sync, automatic conflict resolution, instant local reads.
  • Turso embedded replicas keep SQLite local and sync to edge — conceptually closest to your model.
  • Newer / smaller ecosystems; Android-native (vs Flutter/RN) support varies — verify the SDK fits Kotlin/Room before betting on it.
  • More moving parts than a snapshot doc for what is currently a simple one-partner share.

Appwrite Open source

Open-source BaaS: DB + Auth + realtime + storage. Cloud or self-host (Docker microservices). Native Android SDK.

  • Native Android SDK, REST/GraphQL/WebSocket.
  • Very generous cloud free tier (75k MAU, 10 GB); Pro ~$15/mo per member.
  • Self-host = infra cost only; full control + data residency.
  • Heavier self-host — Docker microservices stack to operate.
  • Smaller community than Firebase/Supabase; fewer Android-specific guides.

PocketBase Self-host

Single ~20 MB Go binary: SQLite + Auth + realtime + file storage + admin UI. Free & open source.

  • Dead-simple — one binary, run anywhere; cheapest possible hosting ($5 VPS).
  • SQLite under the hood — same engine family as Room.
  • Total control + data residency; no per-op billing ever.
  • You own uptime, backups, scaling, security patching.
  • Single-node by design — fine for an indie app, not for huge scale; no official Kotlin SDK (REST is straightforward though).

DIY: managed Postgres + thin API Most control

e.g. Neon / Supabase-Postgres / Cloud SQL behind a small Ktor or Cloud Functions API.

  • Maximum control & portability; pick any region.
  • Cheap at small scale (serverless Postgres free/low tiers).
  • You build & maintain auth, API, sync, offline reconciliation — the most engineering effort of any option.
  • Overkill for a one-partner snapshot. Only worth it if the backend becomes a real product surface.

The takeaway from the numbers

Firebase only becomes "expensive" for this app at the million-family level — and a free baby tracker reaching a million active sharing families is a wonderful problem you'd happily re-architect for. Below that, the absolute dollars are small on every option. Optimize for lock-in, residency, and your time — not for these amounts. The one Firebase number that can hurt at any scale is a runaway-bug bill, which a budget cap neutralizes.

5. The factor that should weigh more than cost: it's baby health data

Feeding, sleep, and especially allergy records are health-adjacent personal data about a minor. If Akachan ships in the EU/UK (or you want to be safe), GDPR treatment and data residency matter more than a few dollars of compute. This is the strongest real argument for eventually moving off Firestore toward Supabase EU, Appwrite EU, or a self-hosted option where you control exactly where data lives and can honour deletion/export requests cleanly.

Mitigation regardless of backend: keep the cloud snapshot minimal (only what the partner needs to see), let Room stay the real store, and make sharing strictly opt-in (which it already is).

6. When to choose what

Choose Firebase if…

  • You want to ship and not touch infrastructure (today's situation).
  • Cost predictability is handled by a budget alert/cap.
  • Realtime + offline polish on Android matters most.

Choose Supabase if…

  • You want flat, predictable bills and real SQL/portability.
  • EU data residency / GDPR posture matters.
  • You're OK rebuilding the auth + sync layer once.

Choose a sync engine if…

  • You want the backend to mirror your local-first Room model directly.
  • Robust offline conflict resolution becomes a feature priority.
  • The Android/Kotlin SDK checks out for your stack.

Choose PocketBase / DIY if…

  • You want total control + lowest cash cost and enjoy running infra.
  • Scale stays indie-sized and you can own backups/uptime.
  • Lock-in avoidance is a top personal value.

7. Suggested path

  1. Now: stay on Firebase. Add a budget alert + hard cap in Google Cloud Billing — this is the single highest-value action and removes the only genuine Firebase risk for you.
  2. Keep the snapshot lean: ensure partner sync writes one summary document per family, not one document per event, and that the partner uses a single listener. This keeps every backend option cheap and makes future migration trivial.
  3. Decouple now, cheaply: your SharingRepository interface already abstracts the backend. Keep all Firestore types behind it so a swap later touches only SharingRepositoryImpl + di/SharingModule.kt.
  4. Trigger a migration only when one of these fires: GDPR/residency requirement, a desire to end lock-in, or unexpected cost. Default target: Supabase (EU); consider a sync engine if offline sync becomes a headline feature.
  5. If you ever self-host (PocketBase/Appwrite), budget for backups, monitoring, and patching — that ongoing time is the real price, not the $5 VPS.
