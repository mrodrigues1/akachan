The big picture

Read this once so the rest makes sense — then you'll know why each step exists.

The app's partner-sharing feature (the read-only baby dashboard you share with your partner) is moving off Firebase and onto Supabase. Two separate services need to be configured, and they reference each other:

Supabase
The backend
Postgres database that stores the shared snapshot + the login system (GoTrue).
Google Cloud
The identity provider
Issues the "Sign in with Google" credentials your app and Supabase both trust.
The app
com.babytracker
Reads 3 secret values at build time to talk to both of the above.

They form a small loop, which is why the order below matters:

  • Google gives you a Web client ID + secret → you paste those into Supabase.
  • Supabase gives you a callback URL → you paste that into the Google Web client.
  • The app needs three values at the end: SUPABASE_URL, SUPABASE_ANON_KEY, and GOOGLE_WEB_CLIENT_ID.
This guide covers only the one-time console/dashboard setup — the part that isn't code. Writing the Kotlin (the Supabase client, the sign-in flow) is separate developer work tracked in the migration plan. When you finish here, a developer has everything they need to wire the app up.

A2

Grab the URL & anon key

Two of the three values the app needs. Copy them somewhere safe now.

  1. Open the API settings
    Project Settings (gear, bottom-left)→API
    Jump to API settings ↗
  2. Copy the Project URL Under Project URL. It looks like: 
    https://abcdefghijklmnop.supabase.co
    This becomes SUPABASE_URL.
  3. Copy the anon / public API key Under Project API keys, copy the key labelled anon public (a long eyJ... string). This becomes SUPABASE_ANON_KEY.
    Copy the anon public key — never the service_role secret key. The service_role key bypasses all security rules; it must never ship in an app.
Paste both into a scratch note for now. In Part C they go into local.properties, which is git-ignored and never committed.
A3

Create the database schema

The 15 tables + security rules that hold the shared snapshot.

The migration defines 15 normalized tables (shares, baby, sessions, sleep_records, bottle_feeds, feed_ops, sleep_ops, and so on) plus Row Level Security (RLS) policies that replace the old firestore.rules. The exact SQL lives in the repo under supabase/migrations/ (authored as part of migration issue 1).

The click-by-click part: how to run SQL

  1. Open the SQL Editor
    SQL Editor (left sidebar)→New query
  2. Paste the migration SQL Open the project's supabase/migrations/*.sql file(s), copy the contents, and paste into the editor. Run each migration file in filename order (they're timestamp-prefixed).
  3. Press "Run" (▶) or Ctrl/Cmd + Enter. You should see Success. No rows returned for a schema script.
  4. Confirm the tables exist
    Table Editor (left sidebar)
    All 15 tables listed in the public schema, with a 🔒 RLS enabled badge on each one.
RLS must stay ON for every table. If any table shows "RLS disabled", its data is world-readable/writable. The migration SQL enables it — don't turn it off.
Prefer the command line? With the Supabase CLI you can run supabase link then supabase db push to apply all migrations at once — but the SQL Editor above needs no install and is the beginner path. The same local CLI stack (supabase start) is what tests/CI use in place of the old Firebase emulator.
A4

Understanding the SQL migration files

What you're actually pasting in the step above — where the files live, how they're named, and what's inside each one.

A migration is a single .sql file that moves the database forward one step. Instead of clicking tables into existence by hand, the whole schema is defined as version-controlled SQL that any environment — your Cloud project, a teammate's, the local CI stack — can replay to reach an identical state. They live in the repo at supabase/migrations/ and are applied in filename order.

Naming & layout

Each file is prefixed with a UTC timestamp so they sort chronologically and never run out of order. Supabase records which have already run (in an internal supabase_migrations.schema_migrations table), so replaying is safe — applied files are skipped.

supabase/
├─ config.toml                          # local stack config; enables the Google provider
└─ migrations/
   ├─ 20260617090000_init_schema.sql      # the 15 tables (design §5)
   └─ 20260617090100_rls_policies.sql     # RLS + helpers + CHECK constraints (design §6)
Splitting schema and security into two files is a convention, not a requirement — one combined file works too. Keeping them separate makes the RLS review (the highest-risk part of this migration) easy to read in isolation.

What each file is responsible for

FileContains
init_schema.sql The 15 normalized tables from design §5 — shares (parent, keyed on the 8-char share code) and its children (baby, sessions, sleep_records, bottle_feeds, milk_bags, inventory, growth, milestones, diapers, doctor_visits, partners, feed_ops, sleep_ops, sleep_prediction), each references shares(code) on delete cascade.
rls_policies.sql Turns on Row Level Security for every table, defines the security definer helper functions (is_share_owner, is_connected_partner) that replace Firestore's isOwner/isConnectedPartner, the per-table SELECT/INSERT/UPDATE/DELETE policies, and the CHECK constraints that mirror isValidFeedOp/isValidSleepOp (action enum, volume_ml 1–5000, length bounds).

A representative slice

You don't author these by hand from this tutorial — that's migration issue 1 in the plan. But seeing the shape makes the "paste and Run" step above concrete. One parent, one child, one ops table, and the security that guards them:

-- init_schema.sql (excerpt) — the design's §5 tables
create table shares (
  code          text primary key,
  owner_uid     uuid not null,
  created_at    timestamptz not null default now(),
  last_sync_at  timestamptz
);

create table baby (
  code          text primary key references shares(code) on delete cascade,
  name          text,
  birth_date_ms bigint,
  allergies     text[]
);

create table feed_ops (
  op_id           text primary key,
  code            text not null references shares(code) on delete cascade,
  author_uid      uuid not null,
  action          text not null check (action in ('create', 'update', 'delete')),
  volume_ml       int check (volume_ml between 1 and 5000),   -- mirrors isValidFeedOp
  notes           text check (char_length(notes) <= 2000)
);

-- rls_policies.sql (excerpt) — the design's §6 rules as RLS
alter table shares  enable row level security;
alter table baby    enable row level security;

-- security definer helper avoids recursive RLS evaluation
create function is_share_owner(share_code text) returns boolean
  language sql security definer stable as $$
    select exists (select 1 from shares
                  where code = share_code and owner_uid = auth.uid());
  $$;

-- any signed-in user may read a share (the code is the capability)
create policy shares_read  on shares for select to authenticated using (true);
create policy shares_write on shares for all to authenticated
  using (owner_uid = auth.uid()) with check (owner_uid = auth.uid());

-- child rows: readable by any authed user, writable only by the share owner
create policy baby_read  on baby for select to authenticated using (true);
create policy baby_write on baby for all to authenticated
  using (is_share_owner(code)) with check (is_share_owner(code));
This is the pattern, not the whole file — the real migration repeats the child-table shape across all 15 tables and adds the connected-partner predicates for feed_ops/sleep_ops (design §6). Copy the actual files from the repo; treat the above as a map for reading them.

Working with migrations from the CLI (optional, for developers)

  1. Create a new migration 
    supabase migration new add_some_table   # creates a timestamped empty .sql file
    Write your SQL into the generated file.
  2. Test it locally first 
    supabase start          # boots the local Postgres/Auth stack in Docker
supabase db reset       # wipes + replays every migration from scratch
    A clean db reset proves the migrations apply in order on an empty database.
  3. Push to the Cloud project 
    supabase link --project-ref <your-project-ref>
supabase db push        # applies any not-yet-run migrations to the remote DB
    This is the CLI equivalent of pasting into the SQL Editor in step A3 — either path works.
Never edit a migration that's already been applied. Once a file has run on any shared database, change it by adding a new migration on top. Editing an applied file makes environments silently diverge — the local db reset sees the new SQL, but the Cloud project already skipped that filename and never re-runs it.

Find your callback URL

You'll paste this into Google in the next part — so keep this tab open.

  2. Note the "Callback URL (for OAuth)" Expand the Google provider. It shows a read-only field like:

Configure the OAuth consent screen

The permission dialog users see. Google won't let you create clients until this exists.

  2. App information

    • User support email: your email.

B3

Create the Web OAuth client

This is the one Supabase uses. Its ID also becomes GOOGLE_WEB_CLIENT_ID in the app.

Why "Web" for an Android app? The native Android Credential Manager flow signs in against a Web client (the serverClientId), and Supabase's GoTrue verifies that same Web client. The separate Android client in step B5 is what lets the on-device account picker trust your app. You need both.
  1. Go to Credentials → Create
    APIs & Services→Credentials→+ Create Credentials→OAuth client ID
    Open Credentials ↗
  2. Application type: Web application Name it Akachan Web (Supabase).
  3. Add the Authorized redirect URI Under Authorized redirect URIs, click + Add URI and paste the Supabase callback URL you copied in step A5: 
    https://<your-project-ref>.supabase.co/auth/v1/callback
  4. Create & copy both values Press Create. A dialog shows your Client ID and Client secret.
    • Client ID (ends in .apps.googleusercontent.com) → this is your GOOGLE_WEB_CLIENT_ID and goes into Supabase.
    • Client secret → goes into Supabase only.
    Save both now. You can re-open the client to see the ID later, but treat the secret like a password — it goes into Supabase and nowhere near the app or git.
B4

Get your app's SHA‑1 fingerprints

The Android OAuth client is tied to your signing keys. You need the SHA‑1 of each.

You need the SHA‑1 for the debug keystore (for development builds) and, before release, the release keystore too. The easiest way — run this from the project root:

# From C:\Users\mathe\StudioProjects\akachan
./gradlew signingReport

Look for the debug variant block and copy the SHA1 line:

Variant: debug
Config: debug
Store: ...\.android\debug.keystore
Alias: AndroidDebugKey
SHA1: A1:B2:C3:D4:...:FF   ← copy this
SHA-256: ...
Alternative with keytool (if you prefer, or CI without Gradle): 
keytool -list -v -alias androiddebugkey \
  -keystore ~/.android/debug.keystore \
  -storepass android -keypass android
Release SHA‑1 matters too. The debug SHA‑1 only signs your local dev builds. Any build signed with a different key (Play App Signing, a CI keystore, a release keystore) needs its own SHA‑1 added to the Android client, or Google sign-in fails silently on that build. If you use Google Play App Signing, grab the SHA‑1 from Play Console → your app → Setup → App integrity and add it too.
B5

Create the Android OAuth client

This lets the on-device account picker return a token for com.babytracker.

  1. Credentials → Create → OAuth client ID Same place as before, but this time choose Application type: Android.
    Credentials→+ Create Credentials→OAuth client ID→Android
  2. Package name Enter exactly: 
    com.babytracker
    (This is the app's applicationId — verified in app/build.gradle.kts.)
  3. SHA‑1 certificate fingerprint Paste the debug SHA‑1 from step B4. Name the client Akachan Android (debug).
  4. Create — press Create. There's no secret for Android clients; the package + SHA‑1 pair is the credential.
  5. Repeat for release When you have a release/Play signing SHA‑1, create a second Android client (same package, release SHA‑1) named Akachan Android (release). You can add it later.
You now have three OAuth clients: one Web (B3) and one or two Android (B5). That's expected and correct.
B6

Enable Google in Supabase

Close the loop: hand the Web client's ID + secret back to Supabase.

  1. Return to the Google provider panel The one from step A5.
    Authentication→Sign In / Providers→Google
  2. Toggle "Enable Sign in with Google" ON
  3. Paste the Web client credentials
    • Client IDs: paste the Web client ID from step B3 (the .apps.googleusercontent.com one).
    • Client Secret (for OAuth): paste the Web client secret from step B3.
  4. Save Press Save at the bottom of the Google panel.
Because the native app uses the ID-token flow, Supabase mainly needs the Web client ID registered here to trust tokens minted for it. Adding the secret enables the standard web redirect flow too — harmless to include and future-proof. If Supabase shows a "Skip nonce check" option, leave it at its default unless a developer instructs otherwise; the app's token request and GoTrue must agree on nonce handling.
C1

Add the three keys to the app

All the console work pays off here — three values into one git-ignored file.

Open (or create) local.properties in the project root and add these three lines alongside the existing sdk.dir=...:

# local.properties — never commit this file (already in .gitignore)
SUPABASE_URL=https://your-project-ref.supabase.co
SUPABASE_ANON_KEY=eyJhbGciOi...your-anon-public-key
GOOGLE_WEB_CLIENT_ID=1234567890-abc...apps.googleusercontent.com
KeyWhere it came from
SUPABASE_URLStep A2 — Project URL
SUPABASE_ANON_KEYStep A2 — anon public key
GOOGLE_WEB_CLIENT_IDStep B3 — the Web client ID
Never commit these. local.properties is git-ignored on purpose. The Gradle build reads them and injects them as BuildConfig fields at compile time. For CI, the same three values are provided as encrypted secrets — not checked into the repo.
The Android OAuth client (B5) has no value to paste anywhere — its whole job is the package + SHA‑1 registration you already did. Only the Web client ID reaches the app.
C2

Build & verify

Confirm the plumbing before you rely on it.

  1. Build the app 
    ./gradlew build
    A green BUILD SUCCESSFUL means the three keys were found and injected. A failure mentioning a missing BuildConfig field or a null property means a key is missing or misspelled in local.properties.
  2. Test sign-in on a real device or emulator with Play Services Run the app, open the sharing screen, and tap Sign in with Google. The native account picker should appear and, after you choose an account, return you signed-in.
  3. Confirm the user landed in Supabase
    Supabase→Authentication→Users
    Your Google account listed as a new user, with provider google. That means the whole loop — app → Google → Supabase — works end to end.

✅ You're done when

  • Supabase project exists and all 15 tables show RLS enabled
  • Google Cloud has 1 Web client + 1–2 Android clients
  • Supabase Google provider is enabled with the Web client ID + secret
  • local.properties has all three keys and ./gradlew build passes
  • A test sign-in creates a user under Authentication → Users
C3

Troubleshooting

The traps people actually hit, and the fix for each.

SymptomCause & fix
Account picker opens then closes instantly, no error, not signed in The build's SHA‑1 isn't registered on an Android OAuth client (B5), or the package name doesn't match com.babytracker. Re-run ./gradlew signingReport, confirm the SHA‑1, and check the Android client. Release builds need the release SHA‑1.
Error 400: redirect_uri_mismatch The Supabase callback URL isn't in the Web client's Authorized redirect URIs (B3), or has a typo. Re-copy it from step A5 exactly.
Sign-in works but Supabase rejects it / invalid nonce Nonce handling mismatch between the app's token request and GoTrue. Check the "Skip nonce check" setting on the Supabase Google provider against how the app requests the token.
Only you can sign in; others get "access blocked" App is in Testing mode and they aren't test users. Add them under the OAuth consent screen → Audience → Test users (B2), or publish the consent screen.
PERMISSION_DENIED / can't read or write data after login An RLS policy is missing or a table has RLS off. Re-check the schema step (A3): every table needs RLS enabled and the migration's policies applied.
BUILD FAILED — null/missing BuildConfig field A key is missing or misspelled in local.properties. Names are case-sensitive: SUPABASE_URL, SUPABASE_ANON_KEY, GOOGLE_WEB_CLIENT_ID.
